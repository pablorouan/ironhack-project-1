🏗️ The Architecture

Classic 3-Tier Design

🌐 Frontend Tier

Vote App: Python/Flask

Result App: Node.js/Express

Public subnet with internet access

⚙️ Backend Tier

Redis: In-memory queue

Worker: .NET Core processor

Private subnet, no internet

🗄️ Database Tier

PostgreSQL: Persistent storage

Private subnet, highly secured

Data Flow: Vote → Redis → Worker → PostgreSQL → Result

🤖 Chapter 5: Ansible Configuration

🐍 The Python Incompatibility Crisis
Have you ever dreamt about automatic updates? Me too.
Amazon Linux instances had the wrong Python version. After going crazy to update them...
🎯 The AMI Plot Twist
I discovered I was running an older Amazon Linux 2 instead of Amazon Linux 2023.
Result: dnf vs yum compatibility chaos! I solved 1 task error, 20 more appeared under the hood. Time to... Rollback!
📚 Critical Lesson Learned:
Always check and set your AMI ID yourself!
Don't blindly trust AI suggestions for infrastructure choices. Also, human reasoning is still something 😂
🎉 Final Victory
I took down my instances and set new ones with a specific AMI ID. Now All services deployed successfully across three tiers.
SSH jump host configuration working perfectly. 👌🏼
Screenshots 5, 6: Ansible provisioning process and the beautiful final deployment 💅🏽

💀 The Biggest Challenge

Knowing When to Stop giving CPR

😱 The PostgreSQL Volume Attempt
Had a perfectly working application.
Decided to add persistent volumes for the database.
Broke the result mechanism mysteriously. Then broke everything else trying to solve it.
🎓 Critical Lessons Learned:
  • Don't break what isn't broken, trying to fix the un-fixable!
  • Test infrastructure changes one by one
  • Keep working backups of everything
  • Last minute add-ons and developments are a big No-No
  • Focus on core requirements before adding extras
🔄 What I'd Do Differently:
Implement changes incrementally
Use Git feature branches properly Activities and development take time. Can't be rushed!
